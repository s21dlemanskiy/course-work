# ADR-0001: Выделить backend в отдельные сервисы

## Статус

Принято

## Дата

2026-06-06

## Контекст

RMA включает разные по характеру backend-функции: управление базой знаний, синхронизацию локальных журналов, онлайн-RAG, STT/TTS, админку и авторизацию. Эти части имеют разные нагрузки, риски и требования к масштабированию. Особенно отличаются Search/RAG Service и Speech Service: они ресурсоемкие и могут деградировать независимо от публикации инструкций или приема OperationLog.

## Решение

Backend MVP проектируется как набор отдельных сервисов:

- API Gateway/Auth;
- Documentation Service;
- Knowledge Sync Service;
- Search/RAG Service;
- Speech Service;
- Operation Log/Sync Service;
- Admin Panel.

Сервисы общаются через API Gateway для внешних запросов и внутренние API/очередь для фоновых задач. Общие хранилища выделяются в слой данных: PostgreSQL, Vector Index, Object Storage и Message Broker.

## Альтернативы

| Альтернатива | Плюсы | Минусы |
|---|---|---|
| Один backend-монолит | Проще запуск, меньше контрактов | Сложнее масштабировать AI-функции и синхронизацию отдельно |
| Микросервисы для всех доменных сущностей | Максимальная независимость | Избыточная сложность для MVP |
| Backend как набор отдельных сервисов | Баланс независимости и управляемости | Нужны contract tests и observability между сервисами |

## Последствия

Положительные:

- Search/RAG и Speech можно масштабировать отдельно.
- Отказ AI-сервисов не блокирует публикацию базы знаний и прием журналов.
- Команды разработки могут развивать контент, синхронизацию и AI независимо.

Отрицательные:

- Нужны явные контракты API.
- Усложняется мониторинг и диагностика.
- Нужно следить за согласованностью версий базы знаний между сервисами.

## Когда пересмотреть

- Если интеграционная сложность окажется выше пользы для MVP.
- Если нагрузка на backend мала и отдельные сервисы не дают практического выигрыша.
- Если один из сервисов требует другого SLA или инфраструктуры.
