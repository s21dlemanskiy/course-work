# 06. Сценарии и потоки

## Основной онлайн-сценарий

Сеть доступна, поэтому техник использует локальный OCR и базу знаний, а также онлайн-помощь RAG или Speech при необходимости.

```mermaid
sequenceDiagram
    actor Technician as Техник
    participant Client as Android Client
    participant LocalDB as Local SQLite
    participant Gateway as API Gateway/Auth
    participant RAG as Search/RAG Service
    participant Speech as Speech Service
    participant LogSync as Operation Log/Sync Service

    Technician->>Client: Сканирует шильдик
    Client->>Client: Local OCR извлекает маркировку
    Client->>LocalDB: Поиск Device и Instruction
    LocalDB-->>Client: Инструкция и чек-лист
    Technician->>Client: Описывает проблему текстом или голосом
    alt голосовой ввод
        Client->>Gateway: Запрос STT
        Gateway->>Speech: Распознать речь
        Speech-->>Gateway: Текст проблемы
        Gateway-->>Client: Текст проблемы
    end
    Client->>Gateway: Онлайн-подсказка по проблеме
    Gateway->>RAG: Найти релевантные фрагменты
    RAG-->>Gateway: Подсказка и ссылки на фрагменты базы знаний
    Gateway-->>Client: Ответ RAG
    Technician->>Client: Выполняет шаги чек-листа
    Client->>LocalDB: Сохраняет MaintenanceJob и OperationLog
    Client->>Gateway: Отправляет outbox-события
    Gateway->>LogSync: Применить события идемпотентно
    LogSync-->>Gateway: Синхронизация подтверждена
    Gateway-->>Client: Ack по operation_event_id
    Client->>LocalDB: Пометить события как synced
```

## Офлайн-сценарий с последующей синхронизацией

Базовый сценарий не зависит от сети: вся база знаний уже находится на Android-устройстве.

```mermaid
sequenceDiagram
    actor Technician as Техник
    participant Client as Android Client
    participant LocalDB as Local SQLite
    participant Gateway as API Gateway/Auth
    participant LogSync as Operation Log/Sync Service

    Technician->>Client: Сканирует шильдик без сети
    Client->>Client: Local OCR извлекает маркировку
    Client->>LocalDB: Поиск Device, Instruction, Checklist
    LocalDB-->>Client: Локальная инструкция
    Client-->>Technician: Показывает, что RAG/STT/TTS недоступны
    Technician->>Client: Выполняет чек-лист
    Client->>LocalDB: Создает MaintenanceJob
    Client->>LocalDB: Добавляет OperationLog в outbox
    loop пока сети нет
        Client->>LocalDB: Хранит события со статусом pending
    end
    Client->>Gateway: При появлении сети отправляет outbox
    Gateway->>LogSync: Применить по idempotency_key
    LogSync-->>Gateway: Ack уже примененных и новых событий
    Gateway-->>Client: Результат синхронизации
    Client->>LocalDB: Обновляет статусы событий
```

## Обновление базы знаний

1. Администратор меняет устройства, инструкции или чек-листы в Admin Panel.
2. Documentation Service сохраняет черновик и после публикации создает новую `knowledge_base_version`.
3. Knowledge Sync Service готовит пакет полной базы знаний или инкрементальное обновление.
4. Android Client периодически проверяет доступную версию.
5. Клиент скачивает обновление, проверяет контрольную сумму и применяет миграцию локального хранилища.
6. Уже начатые операции продолжают ссылаться на старую `instruction_version`.
7. Новые операции используют актуальную версию базы знаний.

## Жизненный цикл MaintenanceJob

```mermaid
stateDiagram-v2
    [*] --> created: операция начата
    created --> device_recognized: шильдик распознан
    created --> manual_device_selected: ручной выбор устройства
    device_recognized --> instruction_loaded: инструкция найдена
    manual_device_selected --> instruction_loaded: инструкция выбрана
    instruction_loaded --> in_progress: чек-лист начат
    in_progress --> paused: техник приостановил работу
    paused --> in_progress: работа продолжена
    in_progress --> completed: чек-лист завершен
    in_progress --> failed: ошибка или невозможность выполнить работу
    completed --> sync_pending: события в outbox
    failed --> sync_pending: события в outbox
    sync_pending --> synced: сервер подтвердил прием
    sync_pending --> sync_failed: ошибка синхронизации
    sync_failed --> sync_pending: повторная отправка
    synced --> [*]
```

## Правила повторов

| Повтор | Правило |
|---|---|
| Повторная отправка `OperationLog` | Сервер применяет событие один раз по `operation_event_id` и `idempotency_key` |
| Повторная загрузка обновления базы знаний | Клиент проверяет `knowledge_base_version` и контрольную сумму |
| Повторный OCR одного шильдика | Создает новую попытку распознавания внутри той же операции |
| Повторный онлайн-RAG запрос | Не меняет состояние операции, сохраняется только как подсказка при необходимости |

## Ошибочные ветки

- Если OCR не уверен в результате, пользователь подтверждает устройство вручную.
- Если локальная база знаний повреждена, приложение блокирует начало новых операций и требует переустановки/повторной загрузки базы знаний.
- Если синхронизация вложения не удалась, `OperationLog` остается в outbox до успешной отправки или ручного решения.
- Если сервер вернул конфликт версии, клиент не меняет уже завершенную операцию, а сохраняет диагностическое событие.
